  - Cause: `ui/tabs/workflows.js` and `site/ui/tabs/workflows.js` only filtered the initially loaded 20-run slice and matched raw trigger source strings, while real persisted runs are frequently `schedule-poll`, `startup`, `manual-sweep`, and `monitor`.
  - Fix: normalized run trigger families (`manual`, `monitor-event`, `event`), improved labels, and auto-loaded older pages while a filter/search is active but the current slice has no matches.
- Stale task slowdown root cause:
  - The `Workflow dispatch regression` tasks were real persisted repo-local tasks with `workspace=null` and `repository=null`.
  - Batch workflow templates in `workflow-templates/task-batch.mjs` were willing to dispatch any `todo` task without requiring workspace/repository ownership, which let invalid backlog items stay eligible and contribute to churn.
  - Fix: both batch templates now require non-empty `workspace` and `repository` before dispatch; mirrored generated defaults were refreshed; regression test added in `tests/workflow-templates.test.mjs`.
- Task cleanup + persistence guard:
  - Found a second bug while removing stale tasks: `task-cli` returned success before the task-store write chain flushed to disk, so deletes could appear successful but not persist.
  - Fixed `task/task-cli.mjs` to await `waitForStoreWrites()` after create/update/delete and added `tests/task-cli.test.mjs` to assert CLI create/delete persist on disk after process exit.
  - Removed the 12 stale `Workflow dispatch regression` task IDs from both repo-local stores:
    - `.bosun/.cache/kanban-state.json`
    - `.bosun/workspaces/virtengine-gh/bosun/.bosun/.cache/kanban-state.json`
  - Post-cleanup verification: both stores now report `count=0` for title `Workflow dispatch regression`.
